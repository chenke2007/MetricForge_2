# 元数据采集与治理闭环 UI 设计

## 目标

打通 MetricForge 阶段 1 的首版治理闭环：用户可以从数据源页面触发元数据采集，采集后查看表结构，并在治理待办页面分配、流转和解决治理事项。

## 范围

本轮覆盖：

- 数据源详情页增加“采集元数据”按钮。
- 调用现有 `/api/metadata/collect/{datasource_id}`，展示采集成功统计或失败原因。
- 元数据空状态增加“去数据源采集”的入口。
- 治理待办详情弹窗增加操作区：
  - 分配负责人。
  - 更新状态。
  - 填写解决说明。
- 操作成功后刷新治理待办列表，保证页面状态和后端一致。

本轮不覆盖：

- 字段语义完整新增/编辑器。
- 元数据采集异步任务队列或进度条。
- Oracle 采集器底层 SQL 逻辑改造。
- API 从 query 参数迁移到 JSON body。
- 权限、登录、审计。

## 架构

首版继续复用现有 FastAPI API 和 Jinja 模板，不新增前端框架。

数据源详情页通过轻量 JavaScript 调用采集 API。按钮进入 loading 状态，接口返回后在页面上显示统计结果；成功后刷新当前页面，使“已采集表”数量和表列表更新。

治理待办页面继续使用已有详情 Modal。Modal 加载待办详情后渲染只读信息和操作表单。分配负责人调用 `/api/governance/{ticket_id}/assign`，状态流转调用 `/api/governance/{ticket_id}/status`。操作成功后刷新当前筛选列表。

## 数据流

元数据采集：

1. 用户进入 `/web/datasources/{ds_id}`。
2. 点击“采集元数据”。
3. 前端 POST `/api/metadata/collect/{ds_id}`。
4. 后端执行 `collect_metadata()`。
5. 前端展示返回的统计信息。
6. 页面刷新，显示已采集表。

治理待办：

1. 用户进入 `/web/governance`。
2. 点击待办标题打开详情 Modal。
3. 前端 GET `/api/governance/{ticket_id}`。
4. 用户填写负责人或状态/解决说明。
5. 前端调用 assign/status API。
6. 成功后刷新当前列表，保留当前 `status` 筛选。

## 错误处理

- 元数据采集失败时，页面显示后端返回的错误摘要，不跳转。
- 治理待办操作失败时，保留 Modal，并显示失败提示。
- 按钮执行期间禁用，避免重复提交。
- 空数据状态提供下一步入口，而不是只展示“暂无数据”。

## 测试

实现阶段采用 TDD：

- 增加数据源详情页测试，断言存在“采集元数据”按钮和采集结果区域。
- 增加元数据空状态测试，断言存在数据源采集入口。
- 增加治理待办页面测试，断言详情弹窗中包含负责人、状态和解决说明操作控件。
- 保留现有页面和 API 测试，确保 `python -m pytest tests/ -q` 全部通过。

## 验收

浏览器验收时至少验证：

- 数据源详情页有“采集元数据”按钮。
- 点击采集按钮时能进入 loading 状态，连接失败时显示错误反馈。
- 元数据空状态有明确入口。
- 治理待办详情弹窗能展示操作区。
- 分配负责人、更新状态后页面刷新并显示新状态。
