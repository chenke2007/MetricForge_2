# AI 智能问数与报表 SQL 开发平台项目需求描述

## 项目名称

AI 智能问数与报表 SQL 开发平台

## 项目概述

本项目拟建设一个基于本地大模型的 AI 智能问数与报表 SQL 开发平台，面向融资租赁业务报表开发、指标查询、经营分析和数据治理场景。项目从 0 开始建设，当前默认不存在独立的数据治理系统、指标平台或指标中台。因此，本系统不仅要提供智能问数和 SQL 辅助开发能力，还要内建轻量级指标治理、元数据语义管理、业务规则管理和治理工单闭环能力，逐步沉淀企业数据语义资产。

平台核心目标是将用户自然语言问题优先转化为结构化指标查询请求，先命中本系统内已沉淀的指标定义、标准报表和可信历史 SQL；只有在标准能力无法覆盖时，才进入受控 Text-to-SQL 辅助生成流程，并通过 SQL 安全校验、Schema Linking、执行计划分析、样例执行和人工复核机制降低错误风险。对于超出系统能力边界的问题，平台不强行回答，而是形成数据治理工单，推动指标、字段、表关系和业务规则持续沉淀。

本项目的本质是建设“轻量指标治理底座 + Text-to-Metrics 优先 + Text-to-SQL 兜底”的智能数据服务平台：让业务用户安全用数，让开发人员高效复用 SQL 资产，让数据治理职责在系统中有入口、有流程、有沉淀，而不是依赖外部已有中台。

## 一、项目背景

当前融资租赁业务报表开发和指标查询依赖开发人员手工理解需求、查找数仓表结构、阅读历史 SQL、确认指标口径并编写 SQL。该过程存在以下问题：

1. 数仓表数量较多，字段含义、枚举值和表关系理解成本高。
2. 历史报表 SQL 分散，复用困难，经验无法持续沉淀。
3. 指标口径依赖人工经验，容易出现不同部门、不同报表口径不一致。
4. 新报表 SQL 开发、调试、优化周期较长。
5. SQL 逻辑正确性、性能合理性缺少系统化自动验证机制。
6. 业务用户的临时问数需求缺少可信响应机制，要么依赖人工排期，要么存在口径误用风险。
7. 数据治理问题常在报表开发时才暴露，缺少从 AI 问数反向驱动治理的闭环。

因此，本项目希望通过本地大模型、系统内建指标治理模块、RAG 知识库、Oracle 元数据采集、Schema Linking、SQL 校验和治理工单机制，建设一个可落地、可验证、可持续沉淀的 AI 智能问数与报表开发平台。一期优先跑通 Oracle 19c 场景，但架构上应避免与 Oracle 强绑定，为后续扩展 Hive、Spark SQL、Doris、ClickHouse、PostgreSQL、MySQL 等数据源预留适配能力。

## 二、落地原则

### 0. 本系统内建轻量数据治理能力

由于项目从 0 开始，不预设已有指标平台、指标中台、数据目录或数据治理系统，本系统需要同步建设以下基础能力：

- 指标治理：指标编码、指标名称、业务定义、计算公式、默认口径、适用维度、负责人、版本和生效状态；
- 维度治理：机构、客户、项目、合同、产品、区域、行业、客户经理等常用维度的定义和层级关系；
- 口径治理：自然月、会计期间、起租日期、投放日期、逾期天数、有效合同等统计口径和过滤规则；
- 元数据语义治理：表中文名、字段业务别名、字段含义、单位、枚举值解释、敏感级别和数据权限；
- 关系治理：主外键、事实表与维表关系、常用 join 路径、跨系统编码映射；
- 模板治理：标准报表、参数化 SQL 模板、历史 SQL 片段和审核通过的查询样例；
- 工单治理：对缺失指标、缺失字段语义、口径冲突、表关系不明等问题形成待办并回流知识库。

项目早期不追求建设完整企业级数据中台，而是建设“够 MVP 使用、能持续扩展”的轻量治理底座，并让每一次智能问数、SQL 生成、人工审核和问题失败都能反向沉淀治理资产。

### 1. Text-to-Metrics 优先

智能问数的优先目标不是让 AI 自由生成 SQL，而是将自然语言需求解析为结构化指标查询请求：

- 指标：如投放金额、起租金额、租赁余额、逾期金额、不良资产余额等；
- 维度：如机构、业务条线、客户经理、行业、区域、产品类型等；
- 过滤条件：如时间范围、合同状态、客户类型、项目状态等；
- 时间口径：如自然月、会计期间、起租日期、投放日期、滚动 7 天等；
- 输出形式：汇总值、趋势、排行、明细、同比/环比等。

当指标、维度、过滤条件和口径均可被本系统已沉淀的治理资产覆盖时，平台应直接调用内建指标查询服务、标准报表或固化 SQL 模板返回结果。

### 2. 可信查询提前固化

对高频、低容错、管理层关注的指标查询，应优先在本系统内沉淀为标准指标、标准查询接口或参数化 SQL 模板。AI 在该场景中主要负责意图识别、指标匹配和参数填充，不直接决定底层 SQL 逻辑。

### 3. 受控 SQL 生成兜底

当需求属于长尾分析、开发辅助或标准指标未覆盖场景时，可进入受控 Text-to-SQL 流程。此时必须限制模型能力边界，要求：

- 一期只允许生成 Oracle SELECT SQL，后续按目标数据源方言生成只读 SELECT SQL；
- 只允许访问授权 schema、表和字段；
- 必须基于召回到的指标口径、表关系和业务规则生成；
- 必须输出字段来源、表关联路径、指标计算逻辑和风险说明；
- 未通过校验或置信度不足时必须进入人工复核。

### 4. 不确定问题进入治理闭环

当系统发现指标缺失、字段语义不清、编码映射缺失、表关系不确定、口径冲突或权限不明确时，不应强行回答。平台应记录用户问题、召回上下文、失败原因和建议治理动作，形成待治理事项。

### 5. 先 Oracle 跑通，后续多数仓扩展

一期以 Oracle 19c 为唯一落地数据源，优先跑通元数据采集、指标治理、历史 SQL 复用、SQL 校验和受控 SQL 生成闭环。系统设计时需要抽象数据源连接、元数据采集、SQL 方言、执行计划、权限校验和样例执行接口，避免在业务逻辑、指标定义和 Prompt 中写死 Oracle 专属实现。

后续扩展 Hive、Spark SQL、Doris、ClickHouse 等数仓时，应通过新增数据源适配器和 SQL 方言适配器完成，而不是重写智能问数和指标治理主流程。

## 三、用户画像与容错度

不同用户使用智能问数的目标和容错度不同，平台需要按角色控制能力边界。

| 用户类型 | 典型需求 | 容错度 | 推荐响应方式 |
| --- | --- | --- | --- |
| 管理层/经营管理用户 | 查询核心经营指标、趋势和排行 | 极低 | 仅返回已治理指标或标准报表结果 |
| 业务人员 | 查询客户、合同、项目、租金等业务明细 | 低 | 优先标准指标和参数化模板，必要时提示口径 |
| 数据分析师 | 临时探索、交叉分析、异常排查 | 中 | 可使用受控 SQL 生成，但结果标记为待验证 |
| 报表/数仓开发人员 | SQL 辅助开发、历史 SQL 复用、性能优化 | 中高 | 提供 SQL 草稿、解释、校验和优化建议 |
| 数据负责人/项目实施人员 | 发现指标、字段、表关系和规则缺口 | 不适用 | 接收治理工单并维护知识资产 |

## 四、四层可信响应架构

平台采用逐级信任模型，根据用户需求命中程度、数据治理成熟度和风险等级自动分流。

### L1 标准服务

面向已在本系统内完成治理和审核发布的标准指标、标准报表或 SQL 模板，提供低延迟、高可靠的查询服务。一期可以只覆盖试点主题域内的少量高价值指标，不要求企业全域一次性治理完成。

- 直接调用本系统内建指标查询服务、标准查询接口或固化 SQL 模板；
- 返回已验证结果，不暴露底层 SQL 给普通业务用户；
- 适用于管理层和高频经营指标查询；
- 要求指标定义、口径、维度、权限和数据来源均已确认。

### L2 可信增强

面向“中间地带”需求：相关表、字段和规则已在本系统中完成基础治理，但尚未形成完整标准指标或报表。

- 使用 RAG 和 Schema Linking 选择相关指标、表、字段和关联路径；
- 在安全边界内生成 SQL 或补全参数化模板；
- 必须经过 SQL 安全校验、语法校验、Explain Plan 和样例执行；
- 仅当 SQL 和口径通过校验、且达到配置的置信度阈值时，结果才可作为业务参考输出；
- 对未审核通过的新 SQL，应展示口径、来源和风险提示，并进入人工确认或沉淀流程。

### L3 探索辅助

面向分析师和开发人员的长尾探索需求。

- 允许模型生成探索性 SQL；
- 所有结果默认标记为“待验证”；
- 必须提示可能存在的口径、关联、权限和性能风险；
- 适合问题分析、报表开发草稿和数据排查，不适合作为正式经营口径直接发布。

### L4 任务受理

当系统无法可靠回答时，自动转为治理任务。

- 记录原始问题、用户角色、业务场景、召回内容、失败原因；
- 识别缺失项：指标缺失、字段语义缺失、枚举值未解释、表关系缺失、规则冲突、权限不明等；
- 生成治理工单，进入数据负责人或项目实施团队待办；
- 治理完成后回流本系统的指标治理模块、知识库和 SQL 模板库。

## 五、数据来源

一期仅接入 Oracle 19c，优先跑通单一数据源闭环。后续新增 Hive、Spark SQL、Doris、ClickHouse 等数据源时，按统一元数据模型补充对应数据源的元数据采集、SQL 方言和执行计划适配。

### 1. 一期：Oracle 19c 数仓元数据

- 表名、字段名、字段类型、字段长度、是否为空；
- 表注释、字段注释、字段业务别名；
- 主键、唯一键、外键、事实表与维表关系；
- 索引、分区、统计信息；
- 表数据量、字段基数、常用过滤字段；
- 字段枚举值及业务含义，例如状态码、产品代码、机构代码等；
- 核心表之间的关联关系和推荐 join 路径。

### 2. 历史已上线报表 SQL

- 已上线经营报表、资产报表、风险报表、财务报表等 SQL；
- 报表名称、报表用途、所属业务条线、使用频率；
- SQL 涉及表、字段、join 条件、where 条件；
- 指标计算逻辑、分组维度、排序规则；
- 历史 SQL 的执行情况、性能问题和优化经验；
- 可复用 SQL 片段和参数化模板。

### 3. 指标字典及指标口径说明

- 投放金额、起租金额、租赁余额、应收租金、实收租金、逾期金额、不良资产余额、拨备金额、IRR、综合收益率等指标；
- 指标唯一编码、中文名称、英文名称、业务别名；
- 指标定义、计算公式、涉及字段；
- 指标适用维度、统计周期和默认时间口径；
- 多口径指标的默认规则和冲突处理方式；
- 是否按自然月、会计期间、起租日期、投放日期等口径统计；
- 是否剔除作废合同、提前结清、冲销、撤销、测试数据等规则；
- 指标版本、发布状态、负责人和生效时间。

### 4. 业务规则和报表模板

- 有效合同、已起租合同、已投放项目、已结清合同等判断规则；
- 逾期、不良、核销、减值、展期、重组等状态判断逻辑；
- 机构、客户经理、业务条线等数据权限规则；
- 投放日报、起租月报、租赁余额分析表、逾期明细表、资产质量报表、经营分析报表、财务核算报表等典型模板；
- 常用查询场景、标准问法、同义词和问题改写规则。

## 六、核心功能模块

本章节描述平台的目标态能力边界，不代表第一轮必须全部实现。实际开发优先级以“八、分阶段迭代路线”为准：阶段 0/1 先完成工程底座、Oracle 元数据采集和轻量指标治理，RAG、Schema Linking、自动 SQL 生成等能力在后续阶段逐步补齐。

### 1. 元数据采集模块

- 一期通过只读 Oracle 账号采集必要元数据；
- 设计统一数据源适配器接口，后续支持 Hive、Spark SQL、Doris、ClickHouse 等数据源；
- 支持定时同步表结构、字段注释、索引、分区和约束信息；
- 采集字段枚举值、空值率、基数、数据量等辅助理解信息；
- 将不同数据源的元数据转化为统一、可检索、可理解的结构化知识；
- 标记敏感表、敏感字段和不可访问对象。

### 2. Schema Linking 与候选对象选择模块

- 根据用户问题识别候选指标、候选表、候选字段和候选关联路径；
- 支持小 schema 直接匹配和大 schema Selector 按需触发；
- 避免将全量 schema 直接塞入模型上下文，降低噪声和 token 消耗；
- 输出结构化候选集、置信度和缺失项；
- 对表关系、字段同义词、编码映射进行显式对齐。

### 3. 历史 SQL 解析模块

- 解析历史报表 SQL；
- 提取 SQL 中涉及的表、字段、join 条件、过滤条件、聚合逻辑；
- 将 SQL 与报表名称、指标口径、业务说明建立关联；
- 支持相似 query 召回、few-shot 示例生成和 SQL 模板复用；
- 识别历史 SQL 中的高频过滤条件、常用关联路径和可沉淀指标。

### 4. 内建指标治理与业务语义层模块

- 管理指标定义、计算逻辑、适用维度和口径规则；
- 建立“指标-维度-字段-表-报表-SQL 模板”的映射关系；
- 支持指标唯一编码、版本、生效状态和负责人；
- 支持多口径指标的默认口径选择和用户确认机制；
- 支持模型在生成 SQL 前明确指标含义和统计规则；
- 为 L1 标准服务提供标准指标查询接口；
- 支持从 Excel 指标字典、历史报表、人工录入和审核通过的 SQL 中逐步沉淀指标资产；
- 一期作为本系统内置模块实现，后续如企业建设独立指标中台，可平滑对接或迁移。

### 5. RAG 知识库模块

- 存储元数据、历史 SQL、指标字典、业务规则、报表模板和治理工单；
- 支持向量检索、关键词检索、结构化检索和 rerank；
- 根据用户需求召回相关表结构、历史 SQL、指标口径和业务规则；
- 将召回内容组装成高质量上下文提供给本地大模型；
- 对召回结果输出来源、置信度和使用边界。

### 6. 自然语言理解与结构化指标请求模块

- 解析用户自然语言问题；
- 识别指标、维度、过滤条件、时间口径、排序方式和输出格式；
- 将用户问题转化为结构化指标查询请求；
- 当存在多口径、多指标歧义时，向用户发起澄清；
- 优先匹配本系统内建指标、标准报表和参数化 SQL 模板。

### 7. SQL 生成模块

- 仅在 L2/L3 场景触发受控 SQL 生成；
- 系统通过 RAG 和 Schema Linking 检索相关上下文；
- 一期调用本地大模型生成 Oracle SELECT SQL，后续按目标数据源生成对应 SQL 方言；
- 输出 SQL、字段来源说明、指标计算说明、使用表说明、关联路径和潜在风险点；
- 对探索性 SQL 默认标记为待验证。

### 8. SQL 自动校验模块

- 校验 SQL 是否为只读 SELECT；
- 禁止 insert、update、delete、merge、drop、truncate、alter 等危险操作；
- 一期校验 Oracle 语法，后续扩展 Hive SQL、Spark SQL 等方言校验；
- 检查是否访问敏感字段或未授权表；
- 进行执行计划分析，一期支持 Oracle Explain Plan，后续按数据源适配；
- 检查是否存在笛卡尔积、异常 join、无条件大表扫描等风险；
- 支持小范围样例执行和结果合理性检查；
- 输出校验结论、风险等级和人工复核建议。

### 9. 反馈、审核与治理沉淀模块

- 保存用户需求、结构化指标请求、召回上下文、生成 SQL、校验结果、人工修改记录；
- 将最终确认的 SQL、指标口径和审核意见回流知识库；
- 对无法回答的问题生成治理工单；
- 支持治理工单分类、负责人分配、状态跟踪和完成后回流；
- 支持持续提升后续 SQL 生成和指标问答质量。

## 七、推荐技术方向

1. 后端服务：Python / FastAPI。
2. 数据库元数据来源：一期 Oracle 19c，后续通过数据源适配器扩展 Hive、Spark SQL、Doris、ClickHouse、PostgreSQL、MySQL 等。
3. 结构化知识库：PostgreSQL / MySQL / SQLite，可根据项目阶段选择。
4. 内建轻量指标治理模块：一期用结构化表承载指标定义、维度、口径、负责人、版本、审核状态和 SQL 模板，后续可独立演进为指标平台或对接企业级指标中台。
5. 向量库：Milvus / Qdrant / pgvector。
6. Embedding 模型：bge-m3 或同类本地 embedding 模型。
7. Rerank 模型：bge-reranker-v2-m3 或同类本地 reranker。
8. 本地大模型：Qwen 系列本地模型或企业实际部署模型。
9. SQL 解析：sqlglot，一期重点适配 Oracle 方言，后续扩展 Hive、Spark SQL 等方言规则。
10. Schema Linking：结合结构化检索、规则匹配、历史 SQL 召回和 LLM Selector。
11. 前端界面：可先做简洁 Web UI，用于录入问题、查看指标匹配、查看召回知识、生成 SQL、查看校验结果和提交治理工单。

### 多数据源扩展设计

为避免后续支持其他数仓时重构主流程，系统需预留以下扩展点：

1. 数据源适配器：统一封装连接配置、连通性测试、权限校验和只读执行。
2. 元数据采集适配器：不同数据源分别实现库、表、字段、注释、分区、索引、约束、统计信息采集。
3. SQL 方言适配器：封装日期函数、分页语法、字符串函数、聚合函数、分区过滤、limit/fetch 等方言差异。
4. 执行计划适配器：Oracle 使用 Explain Plan，Hive/Spark/Doris/ClickHouse 按各自引擎能力实现。
5. 语法校验适配器：根据目标数据源选择对应 SQL parser、dry-run 或 explain 校验方式。
6. 指标物理映射层：指标定义与物理表解耦，同一指标可在不同数据源下维护不同表字段映射和 SQL 模板。
7. Prompt 方言上下文：生成 SQL 前必须明确目标数据源和 SQL 方言，避免混用 Oracle 与 Hive 等语法。

## 八、分阶段迭代路线

本项目从 0 开始建设，范围较大，必须采用分阶段、小步快跑、持续验证的方式推进。每个阶段都应形成可运行、可演示、可验收的结果，避免一开始就同时建设完整指标平台、完整 RAG、完整智能问数和完整治理流程。

总体原则：

1. 先跑通最小闭环，再扩大覆盖范围。
2. 先服务报表/数仓开发人员，再逐步开放给业务用户。
3. 先用结构化检索和规则匹配解决确定性问题，再引入向量检索、rerank 和复杂 Agent。
4. 先沉淀少量高价值指标和 SQL 模板，再扩展为更完整的指标治理体系。
5. 每一阶段都保留人工审核入口，不把未验证结果直接作为正式经营口径。

### 阶段 0：项目骨架与本地可运行底座

目标：先把工程跑起来，形成后续可持续迭代的基础。

范围：

1. 搭建后端基础工程和配置体系。
2. 建立项目目录结构、日志、异常处理、配置文件和 README。
3. 建立结构化数据库表，用于保存元数据、指标、历史 SQL、SQL 校验记录和治理待办。
4. 设计数据源适配器、元数据采集适配器和 SQL 方言适配器接口。
5. 提供最简单的 Web UI 或 API，用于查看系统状态、上传文件和查询已导入资产。

验收标准：

- 项目可本地启动；
- 配置文件可管理 Oracle、模型、知识库、安全规则和目标 SQL 方言；
- 数据库表结构初始化成功；
- README 能说明如何启动和扩展。

### 阶段 1：元数据与指标治理最小闭环

目标：先建设“轻量指标治理底座”，不急于生成 SQL。

范围：

1. 基于数据源适配器实现 Oracle 只读连接。
2. 采集指定 schema 或指定主题域的表、字段、注释、索引、约束和枚举值。
3. 导入指标字典文件，例如 Excel、CSV、Markdown。
4. 建立最小指标治理数据表，管理指标编码、名称、定义、公式、维度、时间口径、负责人、审核状态。
5. 支持人工维护字段业务别名、枚举值解释、常用过滤条件和表关系说明。
6. 提供指标、字段、表关系的查询和维护界面。

验收标准：

- 至少完成 1 个试点主题域的数据元数据采集；
- 至少沉淀 5-10 个高价值指标；
- 每个试点指标能看到定义、口径、涉及字段、负责人和审核状态；
- 能发现并记录字段语义缺失、口径不清、表关系不明等治理待办。

### 阶段 2：历史 SQL 资产沉淀与模板复用

目标：先让系统帮助开发人员复用已有 SQL，降低报表开发成本。

范围：

1. 导入试点范围内的历史 SQL 文件。
2. 解析 SQL 中涉及的表、字段、join 条件、where 条件和聚合逻辑。
3. 将历史 SQL 与指标、报表名称、业务说明建立关联。
4. 支持按指标、表、字段、报表名称、关键词检索历史 SQL。
5. 支持将审核通过的历史 SQL 沉淀为参数化模板。

验收标准：

- 至少导入并解析一批试点历史 SQL；
- 能检索相似 SQL 和复用 SQL 片段；
- 至少沉淀 3-5 个可参数化的 SQL 模板；
- 开发人员可基于模板快速生成报表 SQL 草稿。

### 阶段 3：Text-to-Metrics 与 L1/L2 问数闭环

目标：实现自然语言到结构化指标查询请求，优先命中标准指标和 SQL 模板。

范围：

1. 用户输入自然语言问题后，解析为结构化指标查询请求。
2. 识别指标、维度、过滤条件、时间范围和时间口径。
3. 优先匹配内建指标和 SQL 模板。
4. 对命中 L1 标准服务的问题，直接返回已审核 SQL 或查询结果。
5. 对 L2 可信增强的问题，补全参数化 SQL 模板，并输出口径说明和风险提示。
6. 对无法匹配的问题，生成治理待办。

验收标准：

- 试点主题域内高频问题可被解析为结构化指标请求；
- 已治理指标可以被准确命中；
- 标准模板可以根据用户参数生成 SQL；
- 未命中问题能进入治理待办，而不是强行回答。

### 阶段 4：受控 Text-to-SQL 与 SQL 校验

目标：在明确安全边界内提供 SQL 辅助生成能力。

范围：

1. 引入 RAG 检索，召回指标、字段、表关系、历史 SQL 和业务规则。
2. 引入 Schema Linking，选择候选表、字段和 join 路径。
3. 在 L2/L3 场景下生成目标数据源方言的只读 SELECT SQL，本阶段先仅实现 Oracle 方言。
4. 实现 SQL 安全校验、语法校验、Explain Plan 和小样本执行。
5. 对探索性 SQL 标记“待验证”，进入人工审核后才能沉淀。

验收标准：

- 生成 SQL 仅使用候选表和候选字段；
- DDL/DML 等危险语句能被拦截；
- SQL 能输出字段来源、指标计算逻辑、关联路径和风险提示；
- 审核通过的 SQL 能回流为模板或案例。

### 阶段 5：扩域、评估与治理闭环强化

目标：在试点成功后逐步扩大范围，并建立质量评估体系。

范围：

1. 从 1 个试点主题域扩展到多个主题域。
2. 增加更多指标、维度、表关系和业务规则。
3. 完善治理工单状态流转、负责人分配、审核发布和版本管理。
4. 建立问数准确率、SQL 通过率、人工修改率、命中率、召回质量、平均响应时间等评估指标。
5. 根据真实使用反馈持续优化指标口径、模板和检索策略。
6. 在 Oracle 闭环稳定后，选择 Hive 或其他一个数据源作为第二数据源试点，验证适配器扩展能力。

验收标准：

- 每个主题域都有明确的指标负责人和审核机制；
- 系统能统计问数命中率、失败原因和治理缺口；
- 治理完成后能反向提升后续问数命中率和 SQL 质量；
- 新增数据源时无需重写指标治理、问数路由和审核主流程。

## 十、项目建设重点

这个项目的重点不是简单调用大模型生成 SQL，也不是依赖外部已有数据治理系统，而是从 0 构建“内建轻量指标治理 + 元数据语义管理 + 历史 SQL + 指标口径 + 业务规则 + RAG + Schema Linking + SQL 校验 + 治理闭环”的完整体系。

其中，RAG 和 Schema Linking 需要重点保证以下内容能够被准确召回和选择：

- 与需求相关的业务指标；
- 指标对应的计算公式和统计口径；
- 相关数据源表、字段、注释、枚举值和关联关系，一期重点覆盖 Oracle；
- 相似历史报表 SQL；
- 常用过滤条件和数据权限规则；
- 可能影响 SQL 正确性和性能的风险点；
- 无法回答时应补齐的治理缺口。

项目推进建议采用“先打个样、让 AI 搭把手、死磕硬骨头”的方式：

1. 先打个样：选择数据基础好的窄域，快速做出可用 MVP，用真实业务问题验证价值。
2. 让 AI 搭把手：用 AI 解析历史 SQL、发现字段语义缺失、识别高频关联路径和治理缺口。
3. 死磕硬骨头：集中处理影响智能问数准确性的关键指标、关键表关系、关键编码和权限规则。

## 十一、建议的 RAG 与路由配置方向

建议将 RAG 设计为“结构化检索 + 向量检索 + 关键词检索 + Schema Linking + rerank”的组合机制，而不是单纯依赖向量相似度。

同时建议增加问题路由配置：先判断用户问题属于 L1 标准服务、L2 可信增强、L3 探索辅助还是 L4 任务受理。

## 十二、Prompt 方向

### 1. 自然语言转结构化指标请求 Prompt

```text
你是融资租赁业务指标分析助手。

请将用户问题解析为结构化指标查询请求，不要直接生成 SQL。

用户问题：
{user_requirement}

已知指标：
{retrieved_metrics}

已知维度：
{retrieved_dimensions}

已知业务规则：
{retrieved_business_rules}

输出 JSON，字段包括：
- metrics: 命中的指标编码和名称
- dimensions: 分组维度
- filters: 过滤条件
- time_range: 时间范围
- time_grain: 时间粒度
- time_caliber: 时间口径
- output_type: 汇总、趋势、排行、明细等
- confidence: 置信度
- ambiguity: 需要用户确认的问题
- route_level: L1/L2/L3/L4

如果指标口径不明确，必须输出 ambiguity，不允许自行猜测。
```

### 2. 受控 SQL 生成 Prompt

```text
你是融资租赁数仓 SQL 开发专家。

只有当问题无法通过标准指标或 SQL 模板直接回答时，才允许生成 SQL。

目标数据源：
{target_data_source}

目标 SQL 方言：
{target_sql_dialect}

结构化指标请求：
{structured_metric_request}

相关指标口径：
{retrieved_metrics}

Schema Linking 候选表和字段：
{schema_linking_result}

相关历史 SQL：
{retrieved_sql_examples}

相关业务规则：
{retrieved_business_rules}

必须遵守：
1. 只能生成目标 SQL 方言下的只读 SELECT SQL；一期目标方言默认为 Oracle。
2. 不允许 DDL/DML。
3. 只能使用候选表和候选字段。
4. 必须使用召回到的有效合同、权限和状态过滤规则。
5. 日期口径必须严格遵守指标口径。
6. 如果关联路径不确定，必须停止生成并说明缺失信息。
7. 输出 SQL 后解释字段来源、表关联方式、指标计算逻辑和性能风险。
8. 对 L3 探索性结果必须标记为待验证。
```
